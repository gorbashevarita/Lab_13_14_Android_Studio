# Лабораторная работа №13-14. ViewModel и архитектура Android-приложений
***
## Описание игры

Игра подразумевает следующую логику:
1. Игрок должен отгадать слово из букв, прдставленных в поле
2. Если игрок отгалывает - за правильную попытку анчисляется 20 баллов
3. Если игрок не отгадывает - поле ввода подсвечивается красным
4. Если игрок не может отгадать слово - он его пропускает, в таком случае ему выдается следущее слово, а за прошлое слово не начисляется баллов
5. Всего за игру можно отгадать 10 слов
6. По окончании игры выводтся сообщение о том, сколько слов отгалано и кнопкой "начать зановл"
7. В верхей части приложения отображается на каком пользователь слове и сколько заработал баллов.
***

## Технологии
`Kotlin` - основной язык разработки Android-приложений с лаконичным синтаксисом и поддержкой корутин.
`Jetpack Compose` -  декларативный фреймворк для создания UI в Android на Kotlin, заменяет XML-разметку.
`ViewModel` - компонент Jetpack для управления состоянием приложения, отделяет логику от UI и сохраняет данные при смене конфигурации экрана.
` StateFlow` - инструмент для реактивного управления состоянием UI, передаёт обновления в реальном времени через потоки Kotlin coroutines.

## Контрольные вопросы
#### 1. Что такое ViewModel и зачем он нужен?
 - Решает проблему потери данных при изменении конфигурации экрана — сохраняет состояние приложения.
 - Без ViewModel при повороте экрана: Activity пересоздаётся, данные в Activity/Fragment теряются, состояние UI сбрасывается.
####  2. В чём разница между MutableStateFlow и StateFlow?
- _uiState и uiState: _uiState — приватное поле MutableStateFlow для изменения состояния; uiState — публичное свойство StateFlow (_uiState.asStateFlow()), доступное для UI.
- «Backing property pattern» — паттерн, когда приватное поле (бэкер) управляет состоянием, а публичное свойство предоставляет доступ к нему (без возможности изменения).
####  3. Почему в ViewModel нельзя хранить Context или View?
- Если сохранить ссылку на Activity в ViewModel — при пересоздании Activity старая копия останется в памяти, возникнут ошибки при обращении к UI-элементам удалённой Activity.
#### 4. Чем StateFlow отличается от remember и rememberSaveable?
- remember — сохраняет состояние только в рамках Composable и между рекомпозициями, но сбрасывается при повороте экрана. Для локальных временных данных
- rememberSaveable — как remember, но сохраняет состояние при изменениях конфигурации. ДЛя важных локальных данных
#### 5. Что такое data class в Kotlin и зачем он используется для UI State?
- Автоматически генерируемые методы: equals(), hashCode(), toString(), copy(), componentN().
- Функция copy() нужна для создания нового экземпляра data class с изменёнными полями 
#### 6. Как работает collectAsState() в Compose?
- UI автоматически обновляется, потому что collectAsState() подписывается на StateFlow и перекомпозирует экран при каждом новом значении.
#### 7. Зачем разделять код на пакеты data и ui?
- Поддерживает принципы:
1. разделение ответственности (Separation of Concerns);
2. модульность — легче тестировать и поддерживать код;
3. независимость UI от логики — можно менять интерфейс, не трогая данные.
#### 8. Что происходит с ViewModel, когда приложение полностью закрывается (process death)?
- Данные не сохраняются — состояние игры сбросится.
- Альтернативы для долговременного хранения:
  1. SharedPreferences — для простых данных (счёт, настройки);
  2. Room Database — для сложных данных (история игр, прогресс);
  3. DataStore — современная замена SharedPreferences;
